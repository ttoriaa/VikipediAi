Agent 框架研究：从 LLM 调用器到任务执行基础设施

这一页用于解释 Agent 框架为什么正在成为独立软件层：它们不只是模型调用封装，而是在模型之上补齐编排、工具、记忆、状态、可观测性和治理能力。

研究主结论 1 真正的分水岭，不是“能不能做 demo”，而是“能不能稳定执行长任务”。
研究主结论 2 协议标准化正在把工具调用和 Agent 协作从框架私有能力，变成生态公共接口。
研究主结论 3 未来竞争重点将从 Prompt 质量，转向状态、治理、可观测性与生产部署能力。
Definition

什么是 Agent 开发框架

Agent 开发框架是为了构建基于大语言模型的自主代理系统而设计的软件框架与运行时工具集。它不是单次问答包装，而是让系统具备目标驱动、环境感知、工具调用、多步规划、状态持久化和可治理执行能力。

Difference

它与传统 LLM 应用的区别

  • 传统 LLM 更偏“一次生成答案”。
  • Agent 框架更偏“围绕目标持续执行任务”。
  • 核心差异在于：规划、工具、状态、记忆、可观测性。
Components

五个核心能力组件

  1. Agent 定义层：身份、角色、指令、权限与边界。
  2. 编排引擎：顺序、并行、路由、循环、handoff。
  3. 工具系统：API、数据库、浏览器、代码执行器、企业系统。
  4. 记忆与状态：短期记忆、长期记忆、checkpoint、恢复。
  5. 可观测性：Tracing、Logging、Eval、成本和错误定位。
Protocols

协议与理念演进

  • MCP：标准化 LLM 与工具/数据源的连接方式。
  • A2A：让不同框架构建的 agent 可以互相通信与协作。
  • Context Engineering：从写 prompt 转向管理上下文供应链。
  • HITL 成熟化：从确认/拒绝升级为中断、恢复、分级审批。
Evaluation

如何评估一个 Agent 框架

  • 架构设计：抽象清晰度、表达力、可扩展性。
  • 多 Agent 编排：协作模式、handoff、group workflow。
  • 记忆管理：短期/长期/跨会话管理完整度。
  • 工具与协议：MCP、A2A、自定义工具生态。
  • 可观测性：trace、日志、调试、评估闭环。
  • 部署成熟度：持久化执行、容错、伸缩、K8s/托管。
  • 生态与社区：模型支持、集成广度、文档和活跃度。
  • 治理与安全：权限、审批、审计、数据边界。

Positioning

主流框架一句话定位

  • LangGraph：偏底层 orchestration runtime，强状态、强控制、强持久化。
  • Microsoft Agent Framework：偏生产级多 Agent 框架，强调 workflow、hosting、observability。
  • CrewAI：偏 agents/crews/flows 的业务协作抽象，上手快。
  • OpenAI Agents：偏模型厂商自带 agent 能力层，接入路径短。
  • Dify：偏可视化平台产品，适合低代码工作流落地。
  • AutoGen：历史影响力强，适合作为多 Agent 演进参考系。

结论一：框架层正在独立

Agent 框架正在从“模型调用封装”演变为独立的软件基础设施层，承接执行、治理与可观测性。

结论二：协议与状态是核心壁垒

未来框架差异化会越来越集中在 MCP/A2A 支持、状态恢复、HITL 和多 Agent 编排能力上。

结论三：生产化比 Demo 更重要

真正的企业价值不在于做一个会说话的 agent，而在于交付一个可审计、可持续、可部署的任务系统。

Conclusion

结论

Agent 框架的核心竞争力，不再只是“能不能做一个 demo”，而是谁能在协议互通、状态持久化、HITL、可观测性和生产治理上提供真正可落地的执行基础设施。

判断规则：同名能力先看语境。讲推理能力是模型层；讲规划执行是 Agent 层；讲 IDE/Web 入口和团队工作流是产品层。
